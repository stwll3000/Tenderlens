## Резюме

Успешно реализован переход TenderLens от "дашборда статистики" к "движку поиска прибыльных лотов". Внедрена система Profit Scoring, которая оценивает каждый лот по 6 критериям и выдаёт агрегированный score (0-100).

## Выполненные работы

### 1. База данных (5 новых таблиц)

✅ **suppliers** — поставщики с ИНН, названием, регионом, флагами СМП/РНП
- Поля для обогащения: выручка, число сотрудников (ЕГРЮЛ)

✅ **lot_participations** — участие поставщика в лоте
- Связь lot ↔ supplier (many-to-many)
- Цена заявки, флаг победителя, ранг, причина отклонения

✅ **lot_categories** — справочник категорий
- Маппинг ОКПД2 → niche_slug
- 15 предопределённых ниш

✅ **price_benchmarks** — кэшированные benchmark'и
- Медианные цены по нише/региону/периоду
- Среднее число поставщиков, медианное снижение цены

✅ **lot_scores** — результаты скоринга
- Profit Score (0-100) + 6 компонентов
- JSON-флаги для детализации

✅ **features/niche_mapping.py** (200 строк)
- Маппинг ОКПД2 → niche_slug

✅ **scoring/rigged_detector.py** (200 строк)

- Purity Score (0-1): 0 = явная заточка, 1 = чистый тендер

## Ключевые метрики

### Profit Score — формула

```
Profit Score = 
    margin_signal          × 0.30 +
    competition_signal     × 0.25 +
    captive_signal         × 0.15 +
    timing_signal          × 0.10 +
    spec_purity_signal     × 0.10 +
    customer_health_signal × 0.10
```

Результат: **0-100** (чем выше, тем прибыльнее лот)

### Компоненты (каждый 0-1)

| Компонент | Что измеряет | Как считается |
|-----------|--------------|---------------|
| **Margin** | НМЦ выше медианы рынка | ratio = НМЦ / median_price; sigmoid((ratio-1)*5) |
| **Competition** | Низкая конкуренция | n_eff < 3 → 0.9; n_eff > 10 → 0.1 |
| **Captive** | Заказчик не "captive" | top_supplier_share < 40% → 0.95; > 80% → 0.05 |
| **Timing** | Здоровые сроки | 7-14 дней → 0.95; < 3 дней → 0.2 |
| **Spec purity** | ТЗ без заточки | 0 паттернов → 0.95; 3+ паттернов → 0.05 |
| **Customer health** | Платёжеспособность | in_rnp → 0.0; иначе 0.5 + бонусы |

## Риски и ограничения

### Текущие ограничения
1. **Малый объём данных:** 150 лотов → нужно минимум 50-100k для статистической значимости
2. **Нет протоколов:** final_price почти всегда NULL → нужно запустить парсер
3. **Веса экспертные:** требуется калибровка на исторических данных через 1-2 месяца
4. **Парсер протоколов:** может сломаться при изменении разметки zakupki.gov.ru

### Риски
1. **Блокировка zakupki.gov.ru:** при агрессивном парсинге → используем задержки 2-5 сек
2. **Изменение структуры сайта:** парсер протоколов нужно будет адаптировать
3. **Недостаточная точность:** без калибровки Profit Score может давать ложные сигналы
