## Доступные поля в данных

### Текущие поля (парсятся корректно)
```python
{
    'reg_number': str,          # номер закупки
    'url': str,                 # ссылка на закупку
    'law': str,                 # '44-ФЗ' или '223-ФЗ'
    'purchase_method': str,     # способ закупки
    'status': str,              # статус закупки
    'object_name': str,         # объект закупки
    'customer_name': str,       # заказчик
    'customer_url': str,        # ссылка на заказчика
    'initial_price': float,     # начальная цена (НМЦ)
    'region_code': str,         # код региона (54, 77, 50, 78, и т.д.)
    'region_name': str,         # название региона
    'scraped_at': str           # timestamp парсинга
}
```

### Поля с детальной информацией (парсятся опционально)
```python
{
    'published_date': str,      # дата размещения (DD.MM.YYYY)
    'updated_date': str,        # дата обновления (DD.MM.YYYY)
    'deadline_date': str,       # окончание подачи заявок (DD.MM.YYYY)
    'okpd2_codes': list,        # коды ОКПД2 (массив строк)
    'participants_count': int   # количество участников (пока NULL)
}
```

### Поля, которые НЕ парсятся (требуют доработки)
```python
{
    'final_price': float,       # цена победителя (для расчёта α)
    'winner_name': str          # победитель закупки
}
```

## Ключевые метрики для расчёта

| Метрика | Формула / логика | Статус |
|---|---|---|
| Снижение цены (α) | `(НМЦ − цена победы) / НМЦ × 100%` | ⏳ Ждём final_price |
| Конкурентность | среднее число участников на лот | ⏳ Ждём participants_count |
| Win rate победителя | доля побед одного ИНН у заказчика | ⏳ Ждём winner_name |
| Скоринг ниши | `volume^0.3 × (1/avg_bids) × regularity` | ⏳ Ждём больше данных |
| Флаги аномалий | win_rate > 70%, снижение ровно 0.5%, срок подачи < 3 дней | ⏳ Ждём даты |

## Проблемы и ограничения

### ⚠️ Известные проблемы

**1. Supabase Session Pooler нестабилен**
- Session pooler (порт 5432) часто обрывает соединение
- Решение: попробовать transaction pooler (порт 6543) или прямое подключение
- Альтернатива: работать с JSON-данными напрямую

**2. Парсер не извлекает цены победителей**
- Для расчёта снижения цены (α) нужно `final_price`
- Требуется парсинг страницы протокола закупки
- Это отдельная задача, требует доработки парсера

**3. Нет данных о количестве участников**
- Информация доступна только в протоколе (отдельная вкладка)
- Требует дополнительных запросов к zakupki.gov.ru
- Отложено на будущее

### 🚫 Ограничения zakupki.gov.ru

- Задержки 2-5 сек между запросами (обязательно!)
- Не более 50 страниц за один запуск (риск блокировки)
- User-Agent обязателен
- Retry-логика при ошибках 429, 503

## Правила написания кода

1. **Всегда** используй type hints
2. Логирование через `logging`, не `print`
3. Конфиги — через `.env` + `python-dotenv`, никаких хардкодов
4. Каждый модуль — отдельный файл с docstring
5. Для БД — SQLAlchemy ORM + Alembic для миграций
6. Обработка ошибок: retry-логика для сетевых запросов (tenacity)
7. Тесты — pytest, хотя бы на критические функции аналитики
8. Windows console — используй `io.TextIOWrapper(sys.stdout.buffer, encoding='utf-8')`
